# Embedding and Automating Power BI

## Purpose

This skill covers the Power BI developer and extensibility track: embedding content in custom applications, automating the service programmatically, building custom visuals, and connecting external tools. It helps you pick the correct embedding scenario, understand the token and capacity model at a high level, and find the right automation surface. Keep code high-level and point to the official docs for full samples.

**Prerequisites (from the full-lifecycle plan):** This is a Phase 8 specialized skill (SKILL 23). It depends on `publishing-and-sharing-content` (workspaces, workspace roles) and `administering-and-governing` (tenant settings, capacity). It cross-references `securing-row-level-access` for RLS in embedded scenarios.

## When to use

Use this skill when the task involves any of:
- Embedding a Power BI report, dashboard, tile, paginated report, report visual, or Q&A into a web app or portal.
- Deciding between **embed for your organization** (user owns data) and **embed for your customers** (app owns data).
- Choosing an embedding identity: **service principal** vs **master user**, or **service principal profiles** for multitenancy.
- Automating Power BI via the **REST API** or **PowerShell** (create workspaces, import .pbix, set credentials, refresh, generate embed tokens).
- Building or importing a **custom visual** (`.pbiviz`).
- Connecting **external tools** (Tabular Editor, DAX Studio, ALM Toolkit, Metadata Translator) to Power BI Desktop.

## The two embedding scenarios

This is the first and most important decision. Both scenarios use the Power BI client APIs (JavaScript/TypeScript) to embed content in an `iframe`, and both require a Microsoft Entra token. They differ in who authenticates and who needs a license.

| | Embed for your organization | Embed for your customers |
|---|---|---|
| Also known as | **User owns data** (SaaS) | **App owns data** (PaaS) |
| Audience | Internal users with org accounts | External users (typically ISV's customers) |
| User authentication | Interactive — app users sign in to Microsoft Entra ID with their own credentials | Non-interactive (silent) — your app authenticates with any method; users need no Power BI account |
| Who needs a Power BI license | Each app user | App users need no license; the embedding identity does |
| Embedding identity | The signed-in app user | A reserved **service principal** or **master user** |
| Token used to embed | Microsoft Entra token (used directly to access Power BI) | Microsoft Entra token → used to generate an **embed token** |
| R & Python visuals | Supported (region restrictions apply) | Not supported |
| A SKU support | Not supported on A SKUs | Supported on A, EM, P, and F SKUs |

Full decision tree, identity choice, no-code alternatives, and licensing matrix: see [embedding-scenario-decision-guide.md](embedding-scenario-decision-guide.md).

## Tokens and capacity (high level)

**Two token types** (confirmed in `embed-tokens.md`):
- **Microsoft Entra token** — required for all REST API operations; expires after one hour. Used in *both* scenarios.
- **Embed token** — used only in *embed for your customers*. Generated by the [Generate token](https://learn.microsoft.com/rest/api/power-bi/embed-token/generate-token) REST API. Specifies which content can be embedded and the access level (`View`, `Edit`/`allowEdit`, `Create`).

**Capacity / SKUs** (confirmed in `embedded-capacity.md`): Production embedding requires a capacity. SKU families:
- **A SKU** — Azure *Power BI Embedded* offer; only for *embed for your customers*.
- **P / EM SKU** — Power BI Premium (Office offer).
- **F SKU** — Microsoft Fabric (F64 or greater lets all org users have free licenses for *embed for your organization*).

For development testing you may use **free embed trial tokens** with a Pro license; until you buy a capacity the *Free trial version* banner appears on the embedded report.

## Automation surface

Automation is built on the same APIs the embedding scenarios use.

- **Power BI REST API** — the primary surface. Used to retrieve workspace/item metadata, create workspaces (`groups`), import `.pbix`/`.pbip` files, take over a semantic model, set data source credentials, trigger refresh, manage service principal profiles, and generate embed tokens. Every call needs a Microsoft Entra token.
- **Power BI .NET SDK** (`Microsoft.PowerBI.Api` NuGet) — typed wrapper over the REST API (e.g. `pbiClient.Reports.GetReportInGroupAsync`, `pbiClient.Profiles.CreateProfile`).
- **Power BI client APIs** (`powerbi-client` JavaScript/TypeScript) — browser-side: embed content, set filters/slicers, bookmarks, navigation, layout, and respond to events.
- **PowerShell** — used in the docs for SDK/package and environment commands (e.g. `dotnet add package Microsoft.PowerBI.Api`, `pip3 install`, `npm install`). Use PowerShell to drive the SDK or REST calls in scripts.

Full operation map (provisioning a tenant, ownership, credentials, refresh, profiles) plus the multitenancy guidance: see [automation-surface-map.md](automation-surface-map.md).

## Multitenancy (embed for your customers at scale)

For ISVs embedding to many customers, use the **workspace separation** model — one workspace per customer tenant. The embedding identity **cannot be granted access to more than 1,000 workspaces** (supported limit); the optimal number of workspaces per identity for performance is *exactly one*.

**Service principal profiles** solve this: a single parent service principal creates lightweight per-tenant profiles via the `Profiles` REST API. Profiles are local to Power BI (unknown to Microsoft Entra), are first-class security principals (can own semantic models and data source credentials, hold workspace roles), and isolate tenants. Recommended when you have many workspaces and more than 1,000 application users. See [automation-surface-map.md](automation-surface-map.md#service-principal-profiles).

## Custom visuals (high level)

Power BI ships **core** visuals and lets you **import** visuals from AppSource. When neither fits, web developers can build a custom visual packaged as a single **`.pbiviz`** file (confirmed in `develop-power-bi-visuals.md`). A `.pbiviz` is imported into a report; it can be private (you/your org) or published to **Microsoft AppSource**, where it can also be **certified**. Paid visuals use the *additional purchase may be required* tag and the Licensing API. Building one requires setting up the developer environment, the visual project structure, and the visuals API — point to the developer/visuals docs rather than writing visual code here.

## External tools (high level)

The **External Tools** ribbon in Power BI Desktop launches locally installed, registered tools, passing them the live model's Analysis Services server name, port, and model name so they connect automatically (confirmed in `desktop-external-tools.md`). Categories: semantic modeling, data analysis, miscellaneous, and custom.

| Tool | Use |
|------|-----|
| **Tabular Editor** | Build/maintain tabular models; multi-select property editing, DAX highlighting |
| **DAX Studio** | DAX authoring, query tracing, performance tuning, formatting |
| **ALM Toolkit** | Schema compare/diff/merge for ALM; deploy across environments, keep incremental-refresh history |
| **Metadata Translator** | Localize model captions/descriptions/folders via Azure Cognitive Services |
| **PowerBI.tips – Business Ops** | One-stop installer/registrar for the tools above |

Tools connect to the model's Tabular Object Model (TOM): all metadata is readable; write operations are supported but can cause inconsistencies, so use caution. Full reference, registration, and caveats: see [external-tools-quick-reference.md](external-tools-quick-reference.md).

## Workflow

Copy this checklist when starting an embedding or automation task:

```
Embedding / automation task:
- [ ] Step 1 — Pick the scenario: internal users with their own login -> embed for your organization; external users / no Power BI login -> embed for your customers.
- [ ] Step 2 — If app-owns-data, pick the embedding identity: service principal (recommended) vs master user; many tenants -> service principal profiles.
- [ ] Step 3 — Confirm capacity/SKU for production (A SKU only for app-owns-data; F64+/P/EM/Pro/PPU rules for user-owns-data).
- [ ] Step 4 — Enable tenant settings if using a service principal ("Service principals can use Fabric APIs"); add the identity to the workspace as Member/Admin.
- [ ] Step 5 — Acquire the Microsoft Entra token; for app-owns-data generate the embed token (set access level and, if needed, RLS effective identity).
- [ ] Step 6 — Embed with the Power BI client APIs (config object: type, id, embedUrl, accessToken, tokenType, permissions).
- [ ] Step 7 — For RLS in embedded content, set the embed token's effective identity (cross-ref securing-row-level-access).
- [ ] Step 8 — Move to production: buy a capacity (replaces free trial tokens).
```

## Common pitfalls

| Pitfall | Why it breaks | Correction |
|---------|---------------|------------|
| Using A SKU for embed for your organization | A SKUs only support *embed for your customers* | Use F/P/EM, or Pro/PPU per user |
| Granting one embedding identity access to >1,000 workspaces | Moves you to an unsupported state; ACL lookups degrade exponentially | Use service principal profiles (one profile per tenant) |
| Shipping to production on free trial tokens | *Free trial version* banner stays on the report | Purchase a capacity before production |
| Storing client secret / master-user password in source files | Sensitive credentials exposed | Use a secret store such as Azure Key Vault; prefer service principal over master user |
| Forgetting `tokenType` in the embed config | Org uses `models.TokenType.Aad`; customer uses `models.TokenType.Embed` | Match `tokenType` to the scenario |
| Service principal not enabled in tenant settings | REST API calls fail | Enable *Service principals can use Fabric APIs* and add the SP to the workspace |
| Master user generating an embed token for a paginated (RDL) report | Not supported for RDL | Use a service principal for paginated-report RLS/embedding |
| Editing model metadata via an external tool without care | TOM write operations can cause model inconsistencies | Treat writes as advanced; validate the model in Desktop afterward |

## Success criteria

- The chosen scenario matches the audience (internal vs external) and licensing reality.
- The embedding identity (and profiles, if multitenant) is justified and within the 1,000-workspace support limit.
- Capacity/SKU is correct for production.
- Token handling is correct: Entra token for all REST calls; embed token only for app-owns-data, with the right access level and RLS effective identity.
- Automation uses documented REST API / SDK operations; no invented endpoints.
- Custom-visual and external-tool advice points to the developer docs rather than inventing APIs.
